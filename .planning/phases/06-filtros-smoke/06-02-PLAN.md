<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL search param `arq_clientes` (untrusted) → Supabase `.eq("arquiteto_id", X)` | Usuário pode editar URL e injetar valor arbitrário no param. |
| `clientes` table read | Já protegido por RLS existente (Phase 1/2). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-02-01 | Tampering | URL param `arq_clientes` injetado em `.eq("arquiteto_id", <param>)` | mitigate | Supabase JS parametriza `.eq()` (sem SQL injection). Se param não for UUID válido, Postgres retorna 0 rows ou erro 22P02 — `fetchClientes` faz `if (error) toast.error(...)`. Aceitar que UUID inválido vira "lista vazia" (UX leve, não bug). NÃO validar UUID client-side (over-engineering — RLS + Postgres tipa coluna). |
| T-06-02-02 | Information Disclosure | Listar clientes de outro colaborador via filtro | accept | Admin já tem RLS que permite ler todos os clientes (D-25 Phase 2). Filtro não vaza — só refina o que admin já podia ver. RLS continua a única fronteira. |
| T-06-02-03 | DoS | Refetch a cada mudança de filtro | accept | Filtro só dispara em onSelect (clique do usuário, não digitação). Sem debounce extra necessário. |
</threat_model>
